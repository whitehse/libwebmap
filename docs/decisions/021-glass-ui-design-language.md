# ADR-021: Glass UI design language (host CSS / Canvas)

## Status

Accepted

## Date

2026-07-18

## Context

The demo accumulated parallel dark blues, borders, and magnifier rim colors
inlined in `index.html`, memory HUD CSS, path-list styles, and
`fiber_style.js` `MAG_*` constants. Program design Track 4 **P4.8** asks for
a documented glass-like design language for SPA chrome without pulling UI into
the C core (ADR-014). WebGPU “glass lens” materials remain optional (P4.12).

## Decision

1. **Host-only** design tokens live in:
   - `demo/display/glass_ui.css` — CSS custom properties + component classes  
   - `demo/display/glass_tokens.js` — same values for Canvas2D / JS  
2. **Magnifier chrome** (`MAG_BG`, `MAG_RIM`, …) is derived from
   `GLASS_LENS` in `glass_tokens.js` (single palette).  
3. **Status colors** match `webmap_status_rgba` (`--glass-status-*` and
   `GLASS_STATUS`).  
4. **No libwebmap C/WASM API** for UI tokens. Edgehost SPA and mobile shells
   may copy or link the CSS later.  
5. Document usage in [guides/glass-ui.md](../guides/glass-ui.md).

## Consequences

- Demo HTML links `glass_ui.css`; shell classes (`glass-shell`, `glass-sidebar`,
  `glass-float`, `glass-hud`) replace most hard-coded hex.  
- Token edits require updating CSS + JS together (`GLASS_TOKENS_VERSION`).  
- Path-list DOM generated by JS keeps existing class names but inherits
  tokenized colors from the stylesheet.

## Alternatives considered

| Option | Why not |
|--------|---------|
| CSS-only (no JS tokens) | Canvas magnifier cannot read custom properties portably without layout thrash |
| Full component framework | Too heavy for demo host |
| WGSL glass materials first | Done as opt-in P4.12 (`?glass_gpu=1`); Canvas remains default |
